# ADR-004: 项目定位明确为 Agent 项目 —— 弃用 PipelineOrchestrator 并移除 USE_AGENT_RUNTIME flag

| 属性 | 值 |
|:---|:---|
| 状态 | 已接受 |
| 日期 | 2026-06-30 |
| 决策人 | yuz |
| 关联文档 | `docs/ARCHITECTURE.md`、`resource/docs/ROADMAP.md`、`resource/docs/PRD.md`、`resource/docs/API.md`、`docs/DATABASE.md`、`docs/CHANGELOG.md`（`docs/agent_design.md` 作为快照文档已完成知识迁移并删除） |

## 背景

Agent Runtime Phase 1-3（Phase-Locked Loop / Tool System / Working Memory 持久化）已经实现并稳定运行，但项目权威文档（`ARCHITECTURE.md`、`ROADMAP.md`、`PRD.md` 等）仍沿用旧 Plan-then-Execute Workflow 视角，导致实现与文档脱节。更严重的是，代码中存在两条执行路径：

1. `AgentRuntime`：通过 Tool Calling 驱动的 Phase-Locked ReAct Loop。
2. `PipelineOrchestrator`：硬编码七阶段串行调度的旧 Workflow 引擎。

两者通过 `USE_AGENT_RUNTIME` feature flag 在 `app/tasks/research_task.py` 中切换。这一 flag 使得项目定位模糊：ResearchMind 到底是 Agent 项目还是 Workflow 项目？

用户明确决定：

- 项目定位应为 **Agent 项目**，而非 Workflow 项目。
- `agent_design.md` 作为快照文档，其中 Phase 1-3 的已实现概念必须同步到权威文档；同步完成后删除该快照文档。
- Phase 4-7（Dynamic Planning / Reflection / Long Memory / Multi-Agent）作为高级功能排到后续版本。
- 对 `PipelineOrchestrator` 采用渐进式清理：先标记 deprecated，后续再彻底删除，避免一次性重写大量测试。

## 决策

1. **项目定位：Agentic Research System**
   - `PRD.md` 明确系统本质为 **Agentic Research System**，采用 **Phase-Locked ReAct** 架构。
   - `ARCHITECTURE.md` 以 `AgentRuntime` + Tool System + Working Memory 为核心重新描述系统分层与状态机。
   - `ROADMAP.md` 在 Phase 7 内新增「Agent 演进路线」，把 Agent Runtime Phase 4-7 映射为 v1.5/v2.0 高级功能；Agent Runtime Phase 1-3 作为独立 Phase 5 排期。

2. **移除 `USE_AGENT_RUNTIME` feature flag**
   - 从 `app/config.py` 删除该配置项。
   - 从 `app/tasks/research_task.py` 删除 `if settings.USE_AGENT_RUNTIME` 分支，主入口固定走 `AgentRuntime`。
   - 删除后生产路径唯一、项目定位清晰，不再保留 Workflow/Agent 双入口。

3. **渐进式弃用 `PipelineOrchestrator`**
   - 不立即物理删除 `app/services/pipeline_orchestrator.py`，在模块级注释中明确标记 **DEPRECATED**。
   - 说明保留原因：避免一次性删除导致 2000+ 行单元测试和多个集成测试重写；待 Agent Runtime 完全稳定后再作为独立清理任务移除。
   - 新功能禁止依赖 `PipelineOrchestrator`；既有测试可继续作为 legacy 测试直接调用该模块，但需在文件中标注 legacy 说明。

4. **同步 Agent 概念到权威文档**
   - `ARCHITECTURE.md`：更新 §2 系统分层、§3.1 Step 语义、§3.3 Execution Context（增加 `agent_context`）、§3.4 Step 执行树（增加 ReAct 链示例）、§3.6 SSE 事件（增加 `agent.thought/action/observation`）、§3.7 TaskStateResolver（按 phase 分组判定）、§5.5 Failure Model（增加 `AGENT_LOOP_EXHAUSTED`）。
   - `ROADMAP.md`：新增 §6 Phase 5「Agent Runtime Phase 1-3」；Phase 7 新增 §8.4「Agent 演进路线」，映射 Phase 4-7 到 v1.5/v2.0；总体时间线状态说明补充 Agent Runtime Phase 1-3 已完成。
   - `PRD.md`：更新产品背景与系统本质描述，明确 Agentic Research System 定位。
   - `API.md`：SSE 事件表与示例中补充 `agent.thought` / `agent.action` / `agent.observation`。
   - `DATABASE.md`：更新 `execution_context` 与 `research_steps.input` 字段说明以匹配 Agent 语义。

5. **测试调整**
   - `tests/unit/tasks/test_research_task.py`：移除 `USE_AGENT_RUNTIME` monkeypatch，将 `PipelineOrchestrator` mock 改为 `AgentRuntime.build_default` mock。
   - `tests/integration/test_pipeline_retry.py`：移除 `USE_AGENT_RUNTIME` monkeypatch，标注为 legacy 测试；将通过 `_run_pipeline` 的调用改为直接调用 `PipelineOrchestrator.run()`，保持对弃用模块断点续跑逻辑的覆盖。

## 后果

### 优点

- 项目定位清晰：ResearchMind 是 Agent 项目，不再是 Workflow 项目。
- 生产路径唯一，消除 `AgentRuntime` 与 `PipelineOrchestrator` 双入口打架的问题。
- 权威文档与实现一致，便于面试、验收与后续开发。
- Phase 4-7 作为高级功能明确排期，不阻塞 v1.0 上线。
- 渐进式弃用降低风险，避免一次性大规模测试重写。

### 缺点 / 风险

- `PipelineOrchestrator` 代码仍保留，需要未来独立清理任务彻底删除。
- 既有 Pipeline retry 集成测试改为 legacy 直接调用，不再覆盖 `_run_pipeline` 的 AgentRuntime 端到端 retry 路径；该路径由 Agent Runtime 自身测试覆盖。
- 文档同步后需要全量回归测试，确保没有遗漏的交叉引用或实现偏差。

## 实现偏差

| 设计点（原始计划） | 实际实现 | 原因 |
|:---|:---|:---|
| 本次物理删除 `PipelineOrchestrator` | 仅标记 deprecated，保留代码与测试 | 用户选择渐进方案，避免一次性波及大量测试与集成逻辑。 |
| `USE_AGENT_RUNTIME` 保留为默认 `True` | 从 `app/config.py` 和 `_run_pipeline` 中完全移除 | 用户明确项目定位为 Agent 项目，不再需要 Workflow 回退开关。 |
| Agent 演进路线作为独立 Phase 或 v1.5 章节 | 放入 `ROADMAP.md` Phase 7「迭代优化」内；Agent Runtime Phase 1-3 作为独立 Phase 5 | 用户明确决定「放 phase6」演进路线、Phase 1-3 作为 Phase 5，编号顺延。 |
